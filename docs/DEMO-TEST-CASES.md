## ⚠️ Honesty caveats before you film (read this)

The hackathon validation is explicitly built around *"transparency and honesty."* Two demo ideas are **overclaims in the current code** — the adversarial verifier flagged both `major`. Don't put them on camera as-is:

1. **🔴 "Byte-for-byte deterministic run" (PERF-1) is unproven.** `GEN_PARAMS` uses `temp=0.6 / top_p=0.95 / top_k=20` — sampling is stochastic, made repeatable *only* if the SDK honors the seed bit-for-bit (unverified for `@qvac/sdk@0.12.2`). Note: `promptSha256` and `promptTokens` match for **any** two runs of the same prompt (they're functions of the input) — so matching them proves *nothing*. The only real determinism test is **`generatedTokens` matching across two seeded runs**. Run that test privately first; if it doesn't hold, say "reproducible setup (fixed seed + params)" not "byte-for-byte identical."
2. **🔴 "TurboQuant long-context headline" (PERF-3) is not implemented.** `agent.ts` has an explicit `TODO(turboquant)` and `MODEL_CONFIG` only wires `ctx_size/gpu_layers/device/verbosity` — there is **no KV-cache-quant flag**. Also the whole 3-doc corpus is only **~1,048 prompt tokens**, so raising `QVAC_CTX` to 16384 demonstrates *no* long-context capability (the window is 16× the data) and the truncation warning never fires either way. Either (a) drop the long-context claim until TurboQuant is wired + you have a genuinely large corpus, or (b) demo it honestly as "context-window plumbing works" without the TurboQuant label.

Everything else below is corpus-true and safe to show.
